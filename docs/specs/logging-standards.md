# System & Playback Logging Standard

This document outlines the standard schema and procedures for managing **System Logs** and **Playback Logs** in the Digital Signage system. These logs are critical for debugging, performance monitoring, and proof-of-play auditing.

## 1. System Logs
Used for technical events (errors, warnings, info, debug) generated by the application or devices.

### 1.1 Table Schema: `SystemLogs`
| Field        | Type              | Description                                      |
|--------------|-------------------|--------------------------------------------------|
| LogId        | NVARCHAR(36) PK   | Primary Key (GUID)                               |
| DeviceId     | NVARCHAR(36) NULL | Device generating the log (NULL for server logs) |
| LogType      | NVARCHAR(50)      | Level: `INFO`, `WARNING`, `ERROR`, `DEBUG`       |
| Message      | NVARCHAR(MAX)     | Detailed log message                             |
| Source       | NVARCHAR(100)     | Component/Service name (e.g., 'Player', 'API')   |
| CreatedAt    | DATETIME          | Timestamp (Server Time)                          |

### 1.3 Message Format Standard
To ensure readability and consistency, use the following **Pipe-Separated Key-Value** format for the `Message` field:

```text
[Component] ACTION | Key: Value | Key: Value ...
```

**Examples:**
*   `[PlaylistService] UPDATE | PlaylistID: 1 | Name: 'Morning Promo' | Count: 5`
*   `[AuthService] LOGIN_SUCCESS | User: admin | IP: 192.168.1.10`
*   `[DeviceService] HEARTBEAT | DeviceUUID: 7bb8111f... | Status: ONLINE`

**Rules:**
1.  **Prefix**: Always start with `[ComponentName]`.
2.  **Action**: Use UPPERCASE for the primary action (e.g., UPDATE, LOGIN, ERROR).
3.  **Separator**: Use ` | ` (space pipe space) to separate fields.
4.  **Identifiers**: Explicitly label identifiers as `ID` (for Integers) or `UUID` (for GUIDs).

### 1.2 Stored Procedure: `sp_system_logs`
Handles all system logging operations.

#### Actions:
1.  **INSERT**
    *   **Params**: `@deviceId` (opt), `@logType`, `@message`, `@source` (opt)
    *   **Logic**: Inserts a new record with `NEWID()` and `GETDATE()`.

2.  **GET_FILTERED** (Primary Query for Admin UI)
    *   **Params**:
        *   `@startDate` (DATETIME, opt)
        *   `@endDate` (DATETIME, opt)
        *   `@logType` (NVARCHAR, opt) - Filter by specific type
        *   `@deviceId` (NVARCHAR, opt) - Filter by specific device
        *   `@page` (INT) - Default 1
        *   `@pageSize` (INT) - Default 50
    *   **Logic**: Returns paginated results ordered by `CreatedAt DESC`. Should return a `TotalCount` for pagination UI.

3.  **CLEAR_OLD**
    *   **Params**: `@daysToKeep` (INT)
    *   **Logic**: Deletes logs older than `GETDATE() - @daysToKeep`.

## 2. Playback Logs
Used for "Proof of Play". Records every time a media file starts or completes playing on a device.

### 2.1 Table Schema: `PlaybackLogs`
| Field        | Type            | Description                                  |
|--------------|-----------------|----------------------------------------------|
| LogId        | NVARCHAR(36) PK | Primary Key (GUID)                           |
| DeviceId     | NVARCHAR(36)    | Foreign Key to `Devices`                     |
| MediaId      | NVARCHAR(36)    | Foreign Key to `MediaFiles`                  |
| PlaylistId   | NVARCHAR(36)    | Foreign Key to `Playlists` (Context)         |
| BranchCode   | NVARCHAR(100)   | Snapshot of device's branch at time of play |
| PlayedAt     | DATETIME        | Completion time of playback                  |
| DurationSec  | INT             | Duration played in seconds                   |
| Result       | NVARCHAR(20)    | `success`, `error`, etc.                     |
| ErrorMessage | NVARCHAR(500)   | (Optional) Error details                     |

### 2.2 Stored Procedure: `sp_playback_logs`
Handles proof-of-play recording and reporting.

#### Actions:
1.  **INSERT**
    *   **Params**: `@deviceId`, `@mediaId`, `@playlistId`, `@durationSec`, `@result`, `@errorMessage`
    *   **Logic**: Snapshots `BranchCode` from `Devices` and inserts record. Updates `Devices.LastCheckIn`, `LastPlaybackAt`, and sets Status to `PLAYING`.

2.  **GET_LATEST**
    *   **Params**: `@top` (Default 100)
    *   **Logic**: Returns most recent log entries.

3.  **GET_BY_DEVICE**
    *   **Params**: `@deviceId`, `@top`
    *   **Logic**: Returns most recent logs for a specific device.

4.  **GET_SUMMARY**
    *   **Params**: `@startDate`, `@endDate`
    *   **Logic**: Aggregates play counts and total duration by Media for analytics dashboards. Filters by `PlayedAt`.

5.  **GET_BRANCH_SUMMARY**
    *   **Params**: `@startDate`, `@endDate`
    *   **Logic**: Aggregates activity by Branch.

6.  **GET_EXPORT_DATA** (Primary Query for Reports)
    *   **Params**: `@startDate`, `@endDate`
    *   **Logic**: Returns detailed flat data (including joined names) for CSV export. Filters by `PlayedAt`.

## 3. Best Practices
*   **Indexing**: Ensure `CreatedAt`, `DeviceId`, and `LogType` are indexed for fast filtering.
*   **Retention Policy**:
    *   *System Logs*: Keep for 30 days (auto-purge via nightly job).
    *   *Playback Logs*: Keep for 1 year (or archive to cold storage/CSV before purging).
*   **Proof of Play (Offline Store-and-Forward)**:
    *   Devices MUST NOT rely on a single volatile HTTP POST for playback logs.
    *   **Implementation**: Use `localStorage` to queue logs on the client side.
    *   **Sync Logic**: Attempt to sync pending logs every 15 seconds (Heartbeat) or immediately after a clip ends.
    *   **Verification**: Logs should only be removed from the local queue AFTER the server returns a `success: true` response.
